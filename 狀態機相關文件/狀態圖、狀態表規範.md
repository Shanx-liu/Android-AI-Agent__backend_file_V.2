# LangGraph 狀態機設計文件（最終版）

## 節點清單

| 節點名稱 | 類型 | 說明 |
|---|---|---|
| 開始 | 起點 | 任務啟動 |
| 分析是否缺少具體需求 | LLM 節點 | 解析使用者指令，判斷是否有執行必要的資訊缺失（平台、規格等） |
| 詢問使用者 | Human-in-the-loop | 向使用者提問，有明確說明就直接去LLM分析；無明確說明則進入使用預設值 |
| 使用預設值 | 執行節點 | 使用者未明確說明時，以合理預設值填入缺少的參數，帶入後續 LLM 分析 |
| LLM 分析指令 / 回傳步驟清單 | LLM 節點 | 解析使用者指令（含已補全的參數），產生步驟清單存入 State；每個步驟只含 step_name，不預先生成操作指令 |
| 任務開始通知 | 執行節點 | 通知 APP 端任務即將開始（可觸發 UI 顯示「執行中」狀態）；不執行任何操作、不等待畫面變動，直接進入主迴圈 |
| 讀取 UI Tree | 執行節點 | 透過 AccessibilityService 擷取當前畫面的完整 View 樹，序列化為 JSON 存入 current_ui_tree |
| LLM 生成操作指令 | LLM 節點 | 輸入當前步驟的 step_name 與 current_ui_tree，輸出具體操作指令存入 current_action |
| 判斷是否是敏感操作 | LLM 節點 | 根據 step_name 與 current_action 的實際內容判斷是否為敏感操作，此時 LLM 已知道要對哪個元件做什麼事 |
| 停住並通知使用者 | Human-in-the-loop | 推送通知給使用者，說明即將執行的具體操作內容（含目標元件與動作） |
| 等待使用者確認 | Human-in-the-loop | 等待使用者在通知列按下確認或取消 |
| 發送操作指令 | 執行節點 | 透過 AccessibilityService 對手機執行 current_action |
| 手機截圖後回傳 / 判斷成功與否 | 執行節點 | 截圖後由 LLM 比對截圖與 step_name，判斷當前步驟是否執行成功 |
| 分析失敗原因 / 提供解決方法 | 錯誤處理 | LLM 分析失敗原因，retry_count + 1；重試時回到讀取 UI Tree 重新感知畫面 |
| 更新狀態機 / 執行下個步驟 | 執行節點 | current_step + 1，清除 current_ui_tree 與 current_action，回到讀取 UI Tree 開始下一步 |
| 判斷任務是否執行完畢 | 執行節點 | 比對 current_step 與 total_step 長度，判斷所有步驟是否已完成 |
| 收尾工作 / teardown | 收尾節點 | 寫 log、推通知給使用者、釋放資源 |
| 結束 | 終點 | 所有路徑的唯一出口 |

## 邊（Edge）與條件

```
開始
  └─────────────────────────────→ 分析是否缺少具體需求

分析是否缺少具體需求
  ├── 是 ──────────────────────→ 詢問使用者
  └── 否 ──────────────────────→ LLM 分析指令 / 回傳步驟清單

詢問使用者
  ├── 明確說明 ────────────────→ LLM 分析指令 / 回傳步驟清單
  └── 無明確說明 ──────────────→ 使用預設值

使用預設值
  └─────────────────────────────→ LLM 分析指令 / 回傳步驟清單

LLM 分析指令 / 回傳步驟清單
  └─────────────────────────────→ 任務開始通知

任務開始通知
  └─────────────────────────────→ 讀取 UI Tree（主迴圈入口，第一次感知畫面）

讀取 UI Tree
  └─────────────────────────────→ LLM 生成操作指令

LLM 生成操作指令
  └─────────────────────────────→ 判斷是否是敏感操作

判斷是否是敏感操作
  ├── 是 ──────────────────────→ 停住並通知使用者
  └── 否 ──────────────────────→ 發送操作指令

停住並通知使用者
  └─────────────────────────────→ 等待使用者確認

等待使用者確認
  ├── 確認 ────────────────────→ 發送操作指令
  └── 取消 ────────────────────→ 收尾工作 / teardown

發送操作指令
  └─────────────────────────────→ 手機截圖後回傳 / 判斷成功與否

手機截圖後回傳 / 判斷成功與否
  ├── 成功 ────────────────────→ 判斷任務是否執行完畢
  └── 失敗 ────────────────────→ 分析失敗原因 / 提供解決方法

分析失敗原因 / 提供解決方法
  ├── retry_count < 3 ─────────→ 讀取 UI Tree（重新感知畫面後重試）
  └── retry_count ≥ 3 ─────────→ 收尾工作 / teardown（放棄）

判斷任務是否執行完畢
  ├── 否 ──────────────────────→ 更新狀態機 / 執行下個步驟
  └── 是 ──────────────────────→ 收尾工作 / teardown

更新狀態機 / 執行下個步驟
  └─────────────────────────────→ 讀取 UI Tree（下一步從這裡重新感知畫面）

收尾工作 / teardown
  └─────────────────────────────→ 結束
```

## State 欄位設計

```python
from typing import TypedDict, Annotated
from langgraph.graph.message import add_messages

class Step(TypedDict):
    step_name: str      # 步驟意圖描述，e.g. "點擊搜尋欄位"

class BoundsXY(BaseModel):
    x: int      # 元件中心點 X（node.x + node.width // 2）
    y: int      # 元件中心點 Y（node.y + node.height // 2）

class Action(BaseModel):            # 每次根據 UI Tree 生成的操作指令
    """操作指令細節"""
    action_type: str                # "click" / "set_text" / "scroll" / "global_back"
    resource_id: str | None         # 優先使用，來自 UiNode.resourceId
    bounds_xy: BoundsXY | None      # resource_id 不存在時的 fallback
    input_text: str | None          # 僅 set_text 時使用
    scroll_direction: str | None    # 僅 scroll 時使用："up"/"down"/"left"/"right"

class State(TypedDict):
    user_command: str                        # 使用者原始指令
    again_check: bool | None                 # 是否再詢問使用者一次的決定變數（None = 尚未執行檢查）
    messages: Annotated[list, add_messages]  # 所有對話紀錄（含除錯時的 LLM 對話）
    define_detail: bool | None               # 判斷是否有明確說明
    clarified_params: dict | None            # 補全的參數（詢問使用者或使用預設值後填入）
    total_step: list[Step] | None            # LLM 分析的步驟清單
    current_step: int                        # 目前執行到第幾步
    current_ui_tree: dict | None             # 當前步驟讀取到的 UI Tree（每步執行前更新）
    current_action: Action | None            # LLM 根據 UI Tree 生成的操作指令（執行完後清除）
    last_screenshot: bytes | None            # 最後一張螢幕截圖
    retry_count: int                         # 失敗重試次數（上限 3）
    task_result: str | None                  # 任務結果：success / cancelled / failed
    error_reason: list[str]                  # 每次失敗原因，最多 3 筆（供 teardown 用）
    next_round_hint: str | None              # 分析失敗原因後給下一輪生成操作指令的提示，e.g. "不要用重複的指令"
    is_sensitive: bool | None                # 是否為敏感操作，是敏感操作時為True，否則為False
    sensitive_reason: str | None             # 該操作為敏感操作的原因
    is_confirmed: bool | None                # 使用者確認或取消，確認為True，否則為False
```

### 欄位說明

| 欄位 | 型別 | 初始值 | 說明 |
|---|---|---|---|
| `user_command` | `str` | 必填 | 任務開始時傳入，全程不變 |
| `again_check` | `bool \| None` | `None` | 分析是否缺少具體需求節點的判斷結果；`None` = 尚未執行，`True` = 需再詢問使用者，`False` = 資訊完整可繼續 |
| `messages` | `Annotated[list, add_messages]` | `[]` | LangGraph 自動 append，保留所有 LLM 對話紀錄 |
| `define_detail` | `bool \| None` | `None` | 判斷是否有明確說明，有則為True |
| `clarified_params` | `dict \| None` | `None` | 詢問使用者後填入明確說明的參數，或由使用預設值節點填入預設值；進入 LLM 分析指令時合併進 user_command context |
| `total_step` | `list[Step] \| None` | `None` | LLM 分析完成後才有值，每個元素只含 `step_name`，不預先包含操作指令 |
| `current_step` | `int` | `0` | 每完成一步 +1 |
| `current_ui_tree` | `dict \| None` | `None` | 每步執行前由 AccessibilityService 讀取並更新，步驟完成後可清除 |
| `current_action` | `Action \| None` | `None` | LLM 根據 step_name 與 current_ui_tree 動態生成，執行完後清除 |
| `last_screenshot` | `bytes \| None` | `None` | 每次截圖後更新，teardown 時清除 |
| `retry_count` | `int` | `0` | 每次失敗 +1，達到 3 次進入 teardown |
| `task_result` | `str \| None` | `None` | teardown 時寫入最終結果 |
| `error_reason` | `list[str]` | `[]` | 每次失敗時 append 原因，最多 3 筆，供 teardown 推通知用 |
| `next_round_hint` | `str \| None` | `None` | 分析失敗原因節點寫入，下一輪「LLM 生成操作指令」時作為額外提示，e.g. `"不要用重複的指令"`；步驟成功後清除 |
| `is_sensitive` | `bool \| None` | `None` | 判斷是否是敏感操作節點的判斷結果；`None` = 尚未判斷，`True` = 為敏感操作，`False` = 非敏感操作 |
| `sensitive_reason` | `str \| None` | `None` | 該操作被判定為敏感操作的原因說明；推送通知給使用者時一併顯示；非敏感操作時為 `None` |
| `is_confirmed` | `bool \| None` | `None` | 等待使用者確認節點的結果；`None` = 尚未收到回應，`True` = 使用者確認執行，`False` = 使用者取消 |

## 收尾工作（teardown）說明

所有路徑（正常完成、使用者取消、retry 超限）都會經過 teardown，統一做以下事情：

1. **推通知給使用者** — 根據 `task_result` 產生對應訊息
   - `success`：「任務已完成 ✓」
   - `cancelled`：「任務已取消」
   - `failed`：「任務失敗，已重試 3 次，請手動操作」
2. **寫 Log** — 利用 LangGraph 的 `get_state_history()` 儲存完整執行軌跡
3. **清理 State** — 移除截圖 bytes、current_ui_tree、current_action 等暫存資料
4. **釋放資源** — 通知 AccessibilityService 停止監聽

## 設計原則

- **任務生命週期完整**：APP 端有明確的兩個信號——「任務開始通知」節點發出開始信號，teardown 節點發出結束信號；兩者之間 APP 端可持續顯示「執行中」狀態，生命週期清楚不模糊
- **單一出口原則**：所有結束情境都經過 teardown，再到結束節點
- **補全前置原則**：資訊缺口在進入 LLM 分析之前解決，確保步驟清單生成時 context 完整
- **預設值不擋路**：使用者無明確說明時以合理預設值繼續，不卡住流程
- **感知優先原則**：每個步驟執行前必須先讀取當下的 UI Tree，所有操作指令都基於真實畫面狀態動態生成，不預先寫死
- **敏感判斷後置**：判斷敏感操作在生成 current_action 之後進行，LLM 同時看到 step_name 與實際操作內容，判斷更準確；通知使用者時也能顯示具體操作細節
- **Retry 重新感知**：重試時回到讀取 UI Tree 節點而非直接重送指令，確保 LLM 看到失敗後的最新畫面狀態再決策
- **Retry 上限**：`retry_count` 達到 3 次強制進入 teardown，避免無限循環
- **Human-in-the-loop 完整**：使用者有確認與取消兩條路，Agent 不擅自執行敏感操作
